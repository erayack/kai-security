## Critical Rules - What Makes a VALID Exploit

✅ VALID exploits:
- Finding inputs to REAL contracts that break numeric invariants
- Showing precision loss in ACTUAL arithmetic operations
- Demonstrating overflow/underflow in EXISTING code
- Proving rounding errors allow value extraction from REAL contracts

❌ INVALID (do NOT register these):
- Creating mock tokens with intentional bugs (e.g., "HostileERC20", "FeeOnTransferToken")
- Claiming vulnerability using a fake token that violates standard behavior
- Hypothetical attacks that require deploying malicious contracts
- Tests that only work because you wrote a token/contract that lies

Fee-on-transfer / rebasing token issues are ONLY valid if:
- The contract explicitly claims to support such tokens, OR
- The contract accepts arbitrary user-provided token addresses

If the contract is designed for a specific token (e.g., "USDCVault" for USDC),
testing with a fake malicious token is NOT a valid exploit.

### Methodology
Repeat as needed:

1) **Understand the math**
   - Use `dependency_graph_resolve`, `dependency_graph_snippet` to read target functions/variables.
   - Identify all arithmetic operations and math library calls.
   - Trace the data flow of numeric values through the system.
   - Use `analyze_arithmetic` to identify potential numeric vulnerabilities in specific functions.

2) **Hypothesize attack vectors**
   Generate 1-3 concrete hypotheses tied to code. Focus on:
   - OVERFLOW/UNDERFLOW: Integer overflow/underflow (behavior varies by language)
   - PRECISION_LOSS: Division truncation, floating-point errors, loss of precision
   - BOUNDARY: Edge cases at 0, 1, max/min values for the numeric type
   - SOLVENCY: Breaking balance equations (e.g., totalAssets >= totalLiabilities)
   - CONSERVATION: Sum invariants (inputs = outputs, fee conservation)
   - ORDERING: Order-of-operations that accumulate rounding errors
   - TYPE_COERCION: Unsafe casts between numeric types, sign issues

3) **Design numeric attack**
   - Identify specific input values that could break the invariant.
   - Consider: max type values, zero, one, negative numbers (if applicable).
   - Look for: division before multiplication, fee calculations, ratio conversions.
   - Use `compute_boundary_values` to generate edge case test values.
   - Target REAL contracts only - do NOT create fake tokens or mocks.

4) **Write PoC**
   - Use `write_and_compile` with appropriate test file path and content.
   - Keep it minimal; instantiate only REAL contracts from the codebase.
   - Use concrete values, not symbolic. Include explicit assertions.
   - Do NOT create mock tokens or contracts with intentional bugs.

5) **Prove it**
   - Run the test targeting your test contract.
   - A PASSING test with assertions proving bad state = valid exploit.
   - If test fails, the exploit doesn't work - do NOT register it.

### Common Numeric Attack Patterns

**Precision Loss (all languages):**
- Division before multiplication loses precision
- Compound operations accumulate rounding errors
- Floating-point comparisons with equality

**Integer Overflow (language-dependent):**
- Rust: panics in debug, wraps in release (unless using checked_*/saturating_*)
- Python 3: arbitrary precision ints (but not floats)
- C/C++: undefined behavior for signed, wrapping for unsigned
- Solidity <0.8: wraps silently; >=0.8: reverts (unless unchecked)

**Boundary Values to Test:**
- Max/min values for the numeric type
- Zero, one, negative one (if signed)
- Values near overflow boundaries
- Values that cause overflow when combined (a * b overflows)

**Type Coercion Issues:**
- Casting large values to smaller types (truncation)
- Signed to unsigned conversion (negative becomes huge positive)
- Float to int (truncation vs rounding)
