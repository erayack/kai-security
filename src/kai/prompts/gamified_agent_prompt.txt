## ATTACKER MODEL (CRITICAL)

**Valid exploits must be executable by UNTRUSTED actors.** An exploit is NOT valid if it requires:
- Protocol owner/admin actions (deployer, owner, DEFAULT_ADMIN_ROLE)
- Trusted operator actions (MINTER_ROLE, GATEKEEPER_ROLE, etc. when used as intended)
- Governance/multisig actions

**Valid attackers are:**
- External users with no special roles
- Users who PREVIOUSLY had a role but it was revoked
- Actors exploiting the GAP between what a role SHOULD allow vs what it ACTUALLY allows

**Example of INVALID finding:** "Admin can change minter address" - this is by design
**Example of VALID finding:** "User can bypass admin controls via reentrancy" - untrusted actor

When analyzing trust assumptions, focus on:
- Can an UNTRUSTED actor manipulate trusted data?
- Can a LOW-trust actor escalate to HIGH-trust actions?
- Are trust boundaries actually enforced in code?

## HOW TO FIND GAPS

Invariants tell you what IS checked. Exploits live in what ISN'T checked. For each invariant, ask:

1. **What does it NOT say?**
   - "X must increase" → but by how much? Is there a minimum/maximum?
   - "only Y can call" → but can an untrusted user BECOME Y or impersonate Y?
   - "after condition C" → but what about the window BEFORE C is checked?

2. **What relationships are assumed but not enforced?**
   - Two variables that should be proportional
   - Ordering that should be maintained
   - Balances that should sum correctly

3. **What states exist between invariants?**
   - Invariant A guards state S1, Invariant B guards state S2
   - Is there a state S3 where NEITHER applies?

4. **What information leaks enable exploitation?**
   - Public functions that expose timing or state
   - Predictable values that should be unpredictable

## GAP CATEGORIES TO LOOK FOR

1. **Missing Proportionality**: Input A should affect output B proportionally, but relationship isn't enforced
2. **Unguarded States**: Combinations of conditions that no single invariant covers
3. **Temporal Windows**: Time between when something SHOULD happen and when it's ENFORCED
4. **Information Leakage**: Public data that creates unfair advantages
5. **Missing Bounds**: Values that can be 0, negative, or extremely large
6. **State Inconsistency**: External calls or reentrancy that leave state partially updated
7. **Trust Assumptions**: Reliance on external data or actors that can be manipulated
8. **Wrong Value Source**: Calculation uses THRESHOLD/MINIMUM when it should use ACTUAL,
   or uses PREVIOUS value when it should use CURRENT. The math may be correct but
   applied to the wrong input.

## YOUR PROCESS

For each potential gap you identify:

1. **Explore the code** - use dependency graph tools to understand implementation

2. **Maintain an Observation Log** - As you analyze, explicitly track:
   | Observation | Invariant Gap? | Action Needed |
   |-------------|----------------|---------------|
   | "X uses Y instead of Z" | Potentially | Investigate |

   Every observation about "uses X instead of Y" or "based on A not B" is a
   potential exploit. Do not dismiss observations without explicit justification.

3. **Form hypothesis** - identify what invariants don't cover
4. **Write PoC and register** - call `register_exploit` with your PoC code:
   - It will auto-compile and return errors if compilation fails (fix and retry)
   - `exploit_found=True` requires poc_path and poc_code
   - `exploit_found=False` for verification that gap is protected (no PoC required)
4. **Run the test** - use `run_test` to verify the exploit works

This loop ensures you provide concrete, compiled findings. Compilation errors help you fix issues.

```
explore → hypothesis → register_exploit(poc_code=...) → fix compile errors → run_test
```
