## Test Success Criteria

**IMPORTANT: Always write and compile a test before registering.**

For `exploit_found=true` (vulnerability found):
- Test must PASS
- Assertions prove something bad happened:
  - Attacker extracted value they shouldn't have
  - Unauthorized action succeeded
  - Invariant broken in REAL contract state after exploit sequence

For `exploit_found=false` (invariant holds):
- Test must PASS
- Assertions prove protection works:
  - Unauthorized calls revert as expected
  - State unchanged after failed attack attempt

## Self-Check Before Concluding "Invariant Holds"

When you observe ANY code behavior, explicitly match it against the invariant:

1. **State your observation literally**: "I observed that X uses Y instead of Z"
2. **Re-read the invariant verbatim** - quote the exact rule
3. **Ask**: "Does my observation MATCH what the invariant describes as a violation?"
4. **If yes** → This IS your exploit. Write a PoC proving it.
5. **If no** → Explain specifically WHY your observation differs from the violation criteria

**CRITICAL WARNING**: If you document behavior like:
- "fee increases based on previous claimFee, not msg.value"
- "uses stale value instead of current value"
- "calculation uses X instead of Y"

...and the invariant says the calculation SHOULD use the other value -
**THAT IS THE EXPLOIT**. Do not dismiss your own correct observations.

Many exploits are missed because agents correctly identify buggy behavior in their
analysis but fail to recognize it matches the invariant's violation criteria.

## Critical Rules - What Makes a VALID Exploit

✅ VALID exploits:
- Finding a sequence of calls on REAL contracts that breaks an invariant
- Showing an attacker can extract value using REAL contract logic
- Demonstrating access control bypass in ACTUAL implementation
- Proving a bug in EXISTING code with normal inputs

❌ INVALID (do NOT register these):
- Creating a mock/fake contract that deliberately violates an invariant by design
- Showing an interface CAN be implemented incorrectly (interfaces are just specs)
- Hypothetical attacks requiring attacker-deployed contracts that lie about their behavior
- Tests that only fail because you wrote a malicious mock (e.g., "BadOracle", "HostileToken")

If the invariant references an interface (e.g., IOracle, IERC20), find the ACTUAL usage
in the codebase and test that concrete scenario - do NOT create a fake implementation.

### Methodology
Repeat as needed:

1) **Understand target**
   - Use `dependency_graph_resolve`, `dependency_graph_snippet`, `dependency_graph_neighbors` to locate target functions/variables and understand preconditions/state.
   - Use `dependency_graph_paths` to find reachability from entrypoints.

2) **Hypothesize**
   Generate 1-3 concrete hypotheses tied to code:
   - ACCESS: missing/incorrect role checks
   - ORDERING: missing initialization / state machine step skipping
   - REENTRANCY: external call before state update
   - STATE: inconsistent state updates across functions
   - VALUE_FLOW: wrong value used in calculation (e.g., minimum vs actual, old vs new)
   - SEMANTIC: code does X but invariant expects Y

   **Multi-Interpretation Rule**: If the invariant mentions a value or calculation:
   - List ALL places where that value is read, written, or derived
   - The invariant may apply to ANY of these - check each one
   - "msg.value is used for X" does NOT mean "msg.value is used for Y"

3) **Write PoC**
   - Use `write_and_compile` with appropriate test file path.
   - Keep it minimal; deploy only needed contracts from the REAL codebase.
   - Do NOT create fake/mock contracts that violate invariants by design.

4) **Prove it**
   - Run the test targeting your test contract.
   - A PASSING test with assertions proving bad state = valid exploit.
   - If test fails, the exploit doesn't work - do NOT register it.

5) **Before concluding invariant holds**
   You MUST articulate the counter-case:
   - "The invariant WOULD be violated if [specific scenario]"
   - "I tested this scenario by [what you did]"
   - "The result was [what happened], proving [why invariant holds]"

   If you cannot construct a concrete violating scenario with specific values/calls,
   you have not adequately tested the invariant.

## Lifecycle Sequence Checklist

**Activate when invariant.type is LIVENESS, ORDERING, or targets time-related variables.**

If the invariant involves deadlines, timers, epochs, rounds, or time-bounded behavior:

1. **Identify the time boundary**: Find the deadline variable, any view that returns remaining time, and mutations gated by time.

2. **Test at three critical points**: Pre-deadline, exact deadline, and post-deadline. Verify behavior matches invariant at each.

3. **Verify view/mutation alignment**: If a time view indicates deadline passed, time-gated mutations must behave consistently. Watch for off-by-one errors in boundary operators.

4. **Test reset ordering**: If applicable, verify reset only callable after round ends and obligations preserved.

## Value Source Trace Checklist

**Activate when invariant.type is VALUE_FLOW, ECONOMIC, or FEE_BOUND.**

If the invariant involves fees, prices, or value calculations:

1. **Identify value sources**: Distinguish between actual value (what was paid), threshold value (minimum required), and stale value (previous state).

2. **Trace the calculation**: Which value is used as the base? Document: "Code uses [X] but invariant expects [Y]"

3. **Test with concrete values**: Test at exact threshold and above threshold. Assert exact computed values.
