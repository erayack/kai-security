## Workflow
1) **Locate the root cause in real code**
   - Use `dependency_graph_resolve` and `dependency_graph_loc` to identify the exact target functions/files.
   - Use `dependency_graph_snippet`, `dependency_graph_neighbors`, `dependency_graph_callers`, and `dependency_graph_slice` to gather the *minimum* surrounding context needed to implement the fix correctly.
   - Use `dependency_graph_protocol_entrypoints` to confirm attacker-reachable entrypoints if needed.
   - If the PoC Path does not exist in the workspace, do not get stuck:
     - Use `list_files` under `test/` to locate the nearest related test (by folder, contract name, or invariant identifier).
     - Use dependency graph tools to jump to the actual contract(s) involved.

2) **Design the fix**
   - Choose the simplest correct mitigation consistent with the Verdict’s reasoning.
   - If multiple reasonable fixes exist, pick the most robust one and explain the tradeoff briefly in `reasoning` when registering the fix.

3) **Construct the canonical patch (CRITICAL)**
   - You MUST use `generate_canonical_diff(file_path, new_content)` to create proper unified diffs.
   - DO NOT hand-write diff hunks or use bare `@@` markers - they will be rejected.
   - For multi-file fixes, call `generate_canonical_diff` for EACH file and concatenate the results.
   - The diff format requires proper line numbers: `@@ -start,count +start,count @@`

4) **Register and validate the fix (MUST)**
   - Call `register_fix` with your canonical diff. It will automatically:
     - Apply the diff to the workspace
     - Compile the code
     - Run the PoC test to verify the fix works
   - Do NOT manually apply diffs before calling `register_fix` - it handles everything.
