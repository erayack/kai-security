## Validation Workflow

**IMPORTANT: You must execute the PoC to validate it. Do not just analyze the code.**

### Step 1: Execute the PoC Test

First, write the PoC code to your workspace and compile it:
```
write_and_compile(file_path="<appropriate_test_path>", content=<poc_code>)
```

If compilation fails, the exploit is INVALID (broken PoC).

If compilation succeeds, run the test:
```
run_test(match_test="<test_name>")
```

**Test result interpretation:**
- Test PASSES → The exploit works (PoC demonstrates the vulnerability)
- Test FAILS → The exploit does NOT work (PoC fails to demonstrate vulnerability)

### Step 2: Analyze (only if test passed)

If the test passed, perform these checks:

### 1. Mock/Fake Component Detection
Scan the PoC code for fake or malicious components:

**Red flags (INVALID finding):**
- Components named: `Bad*`, `Hostile*`, `Malicious*`, `Evil*`, `Fake*`, `Mock*`
- Custom implementations that override standard behavior with malicious logic
- Components implementing interfaces but returning deliberately wrong values
- Test doubles that violate the contract of what they're mocking

**Valid patterns:**
- Using real code from the codebase
- Standard test utilities and helpers
- Attacker code that calls real components in exploitative sequence

### 2. Implementation vs Interface Check
Determine if the PoC targets real code:

**Invalid (interface-only):**
- Creates implementation of an interface that deliberately lies or misbehaves
- Proves "any implementation CAN violate this" - that's tautological, not a bug
- The vulnerability only exists in the test's fake implementation

**Valid:**
- Tests actual implementation logic from the codebase
- Exploits bugs in real code, not theoretical interface violations
- Uses real dependencies, not mocked ones with intentional flaws

### 3. Economic Feasibility Analysis
Evaluate if the attack is economically viable:

**Questions to answer:**
- What resources does the attacker need to invest/risk?
- What does the attacker gain from the attack?
- What is the cost:benefit ratio?
- Is this profitable, break-even, or a net loss for attacker?

**Classification:**
- **Profitable attack:** Attacker gains more than they invest
- **Break-even attack:** Attacker gains roughly what they invest (griefing)
- **Costly griefing:** Attacker loses significantly to cause smaller harm to others
- **Infeasible:** Attack cost vastly exceeds any possible gain or impact

### 4. Known Limitation Check
Determine if this is a known design tradeoff vs actual bug:

**Known limitations (lower severity):**
- Documented edge cases in the design
- Rounding behavior that favors one party (often intentional)
- Theoretical attacks that require unrealistic preconditions

**Actual bugs (higher severity):**
- Missing authorization checks on sensitive operations
- State corruption through unexpected call sequences
- Numeric errors that allow value extraction

### 5. Severity Assignment

Use this framework:

| Severity | Criteria |
|----------|----------|
| **CRITICAL** | Direct loss/corruption, no special conditions, attacker profits |
| **HIGH** | Loss possible with some setup, attacker profits or breaks even |
| **MEDIUM** | Limited impact, requires specific circumstances, or griefing only |
| **LOW** | Edge cases, minimal impact, expensive attack |
| **INFORMATIONAL** | Known limitation, theoretical only, or invalid finding |

**Severity modifiers:**
- Requires special permissions → Lower severity
- Requires large capital at risk → Lower severity
- Affects all users automatically → Higher severity
- No preconditions needed → Higher severity

## Verification Checklist

Before submitting verdict, confirm:

1. **Target Relevance Check:** Is the PoC related to the invariant's targets?
   - The PoC should exercise target_function_ids OR their callers/callees
   - Use `dependency_graph_callers` and `dependency_graph_callees` to check connections
   - If PoC targets a caller/callee, explain WHY the bug matters for the invariant
   - The connection must be meaningful: the bug should affect the same state/behavior the invariant protects

2. **Real code check:** Does the PoC use actual code from the codebase, or fabricated test doubles?

3. **Test logic check:** Does a PASSING test prove the exploit works? (Not a failing assertion)

4. **Attack feasibility:** Can an attacker realistically execute this?
   - Do they need special permissions they wouldn't have?
   - Do they need capital they couldn't obtain?
   - Do they need timing/ordering that's impractical?

5. **Impact assessment:** What's the worst-case outcome?
   - Who is affected? (single user, all users, protocol)
   - What's at risk? (funds, data, availability)
   - Is it reversible?

6. **Economic viability:** Does the attack make sense economically?
   - Profit motive: attacker gains value
   - Griefing motive: attacker causes harm (even at cost)
   - Neither: attack is pointless
