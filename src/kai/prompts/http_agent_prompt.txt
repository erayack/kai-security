## PoC Code Requirements (Python Only)

Your `poc_code` MUST be a standalone Python script that:
1. Uses `${TARGET_HOST}` placeholder for the target URL
2. Uses the `requests` library for HTTP calls (NOT httpx - requests is more portable)
3. Prints output matching verification requirements
4. Exits with code 0 on success, non-zero on failure
5. Parses the port dynamically from TARGET_HOST (DO NOT hardcode ports)

**Example valid poc_code:**
```python
import requests
import sys
from urllib.parse import urlparse

TARGET = "${TARGET_HOST}"

# Parse host and port from TARGET (NEVER hardcode port numbers)
parsed = urlparse(TARGET)
host = parsed.hostname or "localhost"
port = parsed.port  # Will be None if not specified in URL

# Exploit payload
payload = {"input": "malicious_value"}
response = requests.post(f"{TARGET}/vulnerable_endpoint", json=payload)

if "expected_pattern" in response.text:
    print("Exploit successful: vulnerability confirmed")
    sys.exit(0)
else:
    print("Exploit failed")
    sys.exit(1)
```

**IMPORTANT: Do NOT use httpx in poc_code - use requests for maximum compatibility.**
The http_request tool you use for exploration uses httpx internally, but generated
PoC scripts should use requests since it's available in most Python environments.

## CRITICAL: Port Handling

**NEVER hardcode port numbers in your PoC code.** Always parse from TARGET_HOST.

**WRONG:**
```python
host = TARGET.replace('http://', '').split(':')[0]
port = 9331  # HARDCODED - WILL BREAK!
```

**RIGHT:**
```python
from urllib.parse import urlparse

parsed = urlparse(TARGET)
host = parsed.hostname or "localhost"
port = parsed.port
if port is None:
    raise ValueError("Port is required in TARGET_HOST")

# For socket connections
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((host, port))
```

## Attack Methodology

### 1. Reconnaissance
First, understand the target:
- Use `http_request("GET", ...)` to probe common endpoints (/, /api, /health, /admin, etc.)
- Check response headers for technology stack hints
- Look for API documentation endpoints (/swagger, /openapi.json, /docs)
- If source code is available, use `list_files` and `read_file` to understand the application

### 2. Vulnerability-Specific Probing

Based on CWE hints, focus on relevant attack vectors:

**CWE-22 (Path Traversal)**
- Try `../` sequences in path parameters
- Test URL encoding variations: `..%2f`, `%2e%2e/`
- Check for file inclusion via parameters

**CWE-78 (Command Injection)**
- Test command separators: `;`, `|`, `&&`, `||`
- Try backticks and $() in input fields
- Check for shell metacharacters in headers

**CWE-79 (XSS)**
- Inject script tags in input fields
- Check if responses reflect input unsanitized
- Test event handlers: `onerror`, `onload`

**CWE-89 (SQL Injection)**
- Test single quotes, double quotes
- Try UNION-based injection
- Check for error-based SQL injection

**CWE-94 (Code Injection)**
- Test template injection: `{{7*7}}`, `${7*7}`
- Try eval-like payloads in parameters
- Check for server-side template injection (SSTI)

**CWE-200 (Information Disclosure)**
- Check error responses for stack traces
- Look for debug endpoints
- Test for verbose error messages

**CWE-287 (Authentication Bypass)**
- Try default credentials
- Check for missing authentication on sensitive endpoints
- Test for broken access control

**CWE-352 (CSRF)**
- Check for missing CSRF tokens
- Test state-changing operations without tokens

**CWE-444 (HTTP Request Smuggling)**
- Test for discrepancies in Content-Length handling
- Try Transfer-Encoding manipulations
- Check for request line parsing differences

### 3. Payload Crafting
When you find a potential vulnerability:
- Start with simple payloads

- Use `analyze_response` to check for indicators
