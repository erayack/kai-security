# Railway Parallel Benchmark Runner

The `evaluation/` harness can run as a fleet of Railway-hosted workers,
each pulling tasks from a shared Postgres queue. This page covers the
provisioning flow end to end.

## Architecture

```
+----------------------------+      +-------------------+
| developer machine          |      | Railway project   |
|                            |      |   "kai-bench"     |
|  evaluation.cli enqueue    +----->+                   |
|  (writes bench_runs +      |      | Postgres (plugin) |
|   bench_tasks rows)        |      |                   |
|                            |      | bench-worker x N  |
|                            |<-----+   replicas        |
|  evaluation.cli watch <id> |      |   (Dockerfile)    |
+----------------------------+      +-------------------+
```

The worker (`python -m evaluation.worker`) is the long-running process
the Docker image starts. It:

1. Connects to Postgres via `DATABASE_URL`.
2. Resolves the adapter named by `BENCHMARK_ADAPTER`.
3. Loops over `claim_next() -> _run_task() -> complete()`.
4. Exits cleanly on `SIGTERM` so Railway can recycle the container.

## One-time setup

### 1. Create the project

```bash
# from the repo root
railway login
railway init             # interactive: pick batuhan's workspace, name = kai-bench
```

### 2. Add a Postgres plugin

In the Railway dashboard for `kai-bench`:

* `+ New` -> `Database` -> `Postgres`.
* Wait for provisioning.
* Copy the resulting `DATABASE_URL` (Railway exposes it as a reference
  variable named `${{Postgres.DATABASE_URL}}`).

### 3. Create the worker service

```bash
railway service create bench-worker
```

Then in the dashboard for `bench-worker` -> Settings -> Build:

* **Builder**: Dockerfile (Railway will pick up `Dockerfile` automatically).
* **Watch path**: `/` (default).

### 4. Configure required env vars

| Variable | Purpose | Source |
|----------|---------|--------|
| `DATABASE_URL` | Shared task queue. | `${{Postgres.DATABASE_URL}}` |
| `OPENROUTER_API_KEY` | LLM credentials. | **You paste it.** |
| `KAI_BACKEND` | Backend selector. | `openrouter` (already in Dockerfile) |
| `BENCHMARK_ADAPTER` | Which adapter to run. | `cybergym` / `bountybench` / etc. |
| `BENCHMARK_CONFIG` | Adapter config JSON. | e.g. `{"server_url":"http://...","data_dir":"/mnt/cg"}` |
| `BENCHMARK_RUN_ID` (optional) | Limit a replica to one run. | Generated by `enqueue`. |
| `BENCHMARK_PIPELINE_ARGS` (optional) | Extra flags as JSON list. | e.g. `["--skip-fixer"]` |
| `BENCHMARK_TASK_TIMEOUT_S` | Wall-clock per task. | Default `3600` |
| `BENCHMARK_POLL_SECONDS` | Sleep between claim attempts. | Default `5` |
| `BENCHMARK_IDLE_EXIT_AFTER` | Auto-exit after N idle seconds. | Default: never |

> Anything that looks like a secret must be set via the dashboard or
> `railway variables set ...` -- **never** baked into the image.

Beyond the required minimum above, see [`.env.example`](../.env.example) at
the repo root for the broader set of optional knobs the worker + RLM core
read — cybergym-specific ones (`KAI_CYBERGYM_HARNESS_URL`), runtime caps
(`KAI_ITER_WALL_CAP`, `KAI_MAX_BLOCKS_PER_ITER`, `KAI_LLM_REQUEST_TIMEOUT_S`),
and worker tuning (`BENCHMARK_POLL_SECONDS`, `BENCHMARK_IDLE_EXIT_AFTER`,
`BENCHMARK_ENV_OVERRIDES`, …). Authoritative consumers live in
`evaluation/worker.py` and `src/ra/core/rlm.py`; `.env.example` documents the
common ones with defaults and short comments.

To rotate the cybergym ngrok URL (free-tier subdomains change on reconnect):

```bash
railway variables --service kai-bench-cybergym-v2 \
  --set KAI_CYBERGYM_HARNESS_URL=https://<new-subdomain>.ngrok-free.app
railway up --service kai-bench-cybergym-v2 --detach
```

### 5. Deploy

```bash
railway up
```

`railway up` uploads the working tree (honouring `.dockerignore`), builds
the image from `Dockerfile`, and rolls out the result. The `.env`,
`output/`, `tests/`, and other heavy / sensitive paths are excluded by
`.dockerignore`, so the image is reproducible across machines.

## Enqueueing work

From your laptop, point `DATABASE_URL` at the Railway Postgres reference
(use `railway variables --service Postgres` to copy it):

```bash
export DATABASE_URL='postgres://...'

uv run python -m evaluation.cli enqueue \
    --adapter cybergym \
    --adapter-config @cybergym.json \
    --limit 10
```

The CLI prints the generated `run_id`. Then watch progress:

```bash
uv run python -m evaluation.cli watch <run_id>
uv run python -m evaluation.cli status <run_id>
```

Both subcommands fall back to the local `output/bench/...` summary when
`DATABASE_URL` is unset, so single-machine usage is unchanged.

## Scaling

To run with N parallel workers:

```bash
railway service scale bench-worker --replicas 4
```

Each replica claims one task at a time. Multiple replicas never race on
the same row because `claim_next` uses Postgres' `FOR UPDATE SKIP
LOCKED`.

## Cost guardrails

* **Cap concurrent tasks per worker.** The worker loop always processes
  one task at a time. If a benchmark adapter needs more, prefer scaling
  replicas over concurrency inside a single container.
* **Set per-task wall-clock caps.** Default `BENCHMARK_TASK_TIMEOUT_S =
  3600`. Lower it during development to fail fast.
* **Set Railway resource limits.** From the dashboard -> Settings ->
  Resources, cap memory (2 GB is enough for one CyberGym task) and
  CPU (1 vCPU).
* **Auto-exit when idle.** Set `BENCHMARK_IDLE_EXIT_AFTER=300` to have
  workers exit after 5 minutes of empty queue; combined with
  `restartPolicyType: ON_FAILURE` you avoid paying for idle containers.
* **Use `BENCHMARK_RUN_ID`** to scope a replica to a single run so a
  runaway batch can't pull tasks from an unrelated run.

## Troubleshooting

* **Worker can't reach Postgres.** Check that `DATABASE_URL` is set via
  the `${{Postgres.DATABASE_URL}}` reference, not pasted verbatim --
  pasting tends to omit `sslmode=require`.
* **All tasks marked `failed` with `worker_exception`.** Check
  `BENCHMARK_CONFIG` parses as JSON, and the adapter's required paths
  (e.g. CyberGym's `data_dir`) exist inside the image. Persistent state
  needed at runtime must come from a Railway volume or be baked in at
  build time.
* **`HEALTHCHECK` keeps failing.** Usually means the build copied the
  wrong source tree (missing `evaluation/`). Verify `.dockerignore`
  isn't excluding it.

## Where you provide secrets

Only one secret is mandatory: `OPENROUTER_API_KEY`. Set it in the
Railway dashboard for `bench-worker` (Service -> Variables). Optional
secrets such as `JINA_API_KEY` follow the same path.
